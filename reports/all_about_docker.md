# All About Docker: Introduction, CI/CD Engine Integration, and Mac Installation

This report is designed for developers who are new to containerization. It explains what Docker is, why it is critical for this CI/CD Engine project, and how to install and configure it on a macOS system.

---

## 🐳 Part 1: What is Docker?

### The Core Problem: "It Works on My Machine!"
In software development, applications often fail when moved from a developer's laptop to a server because of differences in the environment (e.g., different Node.js versions, missing libraries, or OS-specific bugs). 

**Docker** solves this by packaging an application and all its dependencies (libraries, runtimes, configurations) into a single, standardized unit called a **Container**.

### Containers vs. Virtual Machines (VMs)
*   **Virtual Machines**: Each VM includes a full copy of an operating system, the application, necessary binaries, and libraries - taking up tens of GBs. VMs are slow to boot and consume massive resources because they run a virtualized hypervisor.
*   **Containers**: Containers share the host operating system's kernel. They share resources dynamically and isolate application processes from the rest of the host. They are lightweight (typically MBs in size) and boot up in milliseconds.

```
+-----------------------------------+     +-----------------------------------+
|          Virtual Machines         |     |             Containers            |
+-----------------------------------+     +-----------------------------------+
|  [App A]   [App B]   [App C]      |     |  [App A]   [App B]   [App C]      |
|  [OS A]    [OS B]    [OS C]       |     |  [Libs]    [Libs]    [Libs]       |
|  [Hypervisor (Hardware Virtual)]  |     |  [     Docker Engine (OS Shared) ]|
|  [Host Operating System        ]  |     |  [Host Operating System          ]|
+-----------------------------------+     +-----------------------------------+
```

### Essential Docker Concepts
1.  **Docker Image**: A read-only template containing instructions for creating a Docker container. Think of it as a blueprint or a class in Object-Oriented Programming (OOP). E.g., a `node:20-alpine` image contains a lightweight Linux OS with Node.js 20 preloaded.
2.  **Docker Container**: A runnable instance of an image. Think of it as an object/instance instantiated from the blueprint. Multiple containers can run simultaneously from a single image.
3.  **Docker Registry**: A storage hub where Docker images are uploaded and shared (e.g., **Docker Hub**).
4.  **Volume**: A mechanisms to persist data generated by and used by Docker containers, or mount local files from the host system directly inside a container.

## 🏗️ Part 2: Why is Docker Needed in This CI/CD Engine?

Our CI/CD Engine needs to clone user code and run builds or tests (e.g., `npm install && npm test`). Executing this code directly on our host server is dangerous and impractical. Docker provides three major benefits:

### 1. Security Isolation (Sandboxing)
If a user registers a repository containing malicious code (e.g., `rm -rf /` or a script that steals environment keys), running it directly on your server would crash it or leak secrets. By running the code inside a **Docker Container**, the code is sandboxed. It has no access to the host's files, processes, or environment variables unless explicitly permitted.

### 2. Dependency Isolation
One repository might require Node.js v14, while another requires Node.js v22. If we run tests on the host, we would have to manage multiple node versions and constantly switch between them. With Docker, we boot a container using a Node.js v14 image for the first repo, and a Node.js v22 image for the second.

### 3. Clean Runtimes (Ephemeral Environments)
Each build should start with a clean slate to prevent files from previous builds from affecting the outcome. With Docker, the worker will:
1. Spin up a fresh container.
2. Mount the cloned code.
3. Run the commands.
4. Immediately destroy (delete) the container.

## 🛠️ Part 3: How to Install Docker on macOS

There are two primary ways to install Docker on Mac: using **Docker Desktop** (GUI + CLI) or using **Homebrew**.

### Method A: Install via Docker Desktop (Recommended)
1. Go to the official download page: [Docker Desktop for Mac](https://www.docker.com/products/docker-desktop/).
2. Choose the correct installer based on your Mac processor:
   * **Mac with Apple Silicon** (M1, M2, M3, M4 chips)
   * **Mac with Intel chip**
   * *(You can verify your chip by clicking the Apple icon  in the top-left corner -> "About This Mac" -> "Chip/Processor")*
3. Open the downloaded `.dmg` file and drag **Docker** to the **Applications** folder.
4. Launch **Docker Desktop** from your Applications folder.
5. Accept the service agreement and follow the setup wizard. You will see a Docker icon (a whale carrying shipping containers) in your top menu bar.

### Method B: Install via Homebrew
If you use the Homebrew package manager, run this command in your terminal:
```bash
brew install --cask docker
```
Once installed, open your Applications folder and launch **Docker Desktop** manually to initialize the hypervisor.

## 🧪 Part 4: Verification & Commands

Once Docker Desktop is running, open your terminal and run the following tests:

1.  **Check version** to ensure the command-line tool is installed:
    ```bash
    docker --version
    ```
    *Expected Output:* `Docker version 24.x.x, build ...`

2.  **Test Container Execution** by running the official "Hello World" test container:
    ```bash
    docker run hello-world
    ```
    *What happens under the hood:*
    * Docker checks if the `hello-world` image is on your laptop.
    * Since it isn't, it downloads (pulls) the image from Docker Hub.
    * It spins up a container, prints a confirmation message, and terminates.

3.  **View Running Containers**:
    ```bash
    docker ps -a
    ```
    *Expected Output:* You should see a list containing the terminated `hello-world` container.

## 🔌 How Our Project Will Talk to Docker

In **Week 3**, the backend worker won't use the terminal directly to run Docker commands. Instead, it will use a Node.js library called **`dockerode`**. 

`dockerode` communicates directly with the **Docker Engine API** via a special file called the Unix Socket: `/var/run/docker.sock`. This socket is a secure API gateway on your laptop that allows our backend code to request actions like "create container", "run npm test inside it", and "return the test log output".
